Cybersecurity FAQ's

How do I prevent getting hacked?

+

There are many things you can do to prevent getting hacked. Use a random password generator or random passphrase (ex. &AzebraisaAstripedfunhorseIthinkitsCUTE&) Be careful where you click! Be on the lookout for phishing scams. Only make online purchases from reputable vendors with website URL starting with https://. Turn on multifactor identification. Change passwords regularly. Don't use the same password for multiple logins. Better yet, use a password manager. Keep your apps and OS updated. Only download reputable apps from app store or Google Play; check reviews. Delete old accounts you no longer use. Encrypt the data on your phone. Use a reputable antivirus software. Turn off bluetooth when not in use. If you have important or sensitive files, keep them backed up in a separate server, and don't disclose the location to untrusted parties.

How can I create a ransomware attack plan for my small business?

+

  • Backup Everything! Practicing the 3-2-1 backup rule (keep at least three copies of your data, and store two (2) backup copies on different storage media, with one (1) of them located offsite) is a common approach to keeping your data safe in almost any failure scenario.
  • Ensure your remote workers secure their remote access.
  • Create a ransomware action plan. Walk your team through a simulation, so you know exactly what actions you'll take in the event of an attack. You can even hire a cybersecurity consultant or firm to run a threat analysis, help you shore up vulnerabilities, and develop an action plan in case it's ever needed.
  • Hands down best advice: Prevention is the best medicine.

For Developers: What are secure coding practices?

+

There are practices during the development phase of an application designed to minimize security vulnerabilites. The list could get quite long, so I'll include only a basic overview here. You can find more info on www.owasp.org.

  • Validate data from unstrusted sources (such as external databases)
  • Validate all input against a 'safelist' of allowed characters
  • Implement character escaping
  • Contextually output encode data returned to the client that originated outside the applications' trust boundary
  • Require authentication for all non-public pages
  • Authentication failers should not indicate which part of the authentication was incorrect
  • Use only HTTP POST requests to transmit authentication credentials
  • Only send non-temporary passwords over an encrypted connection
  • Enforce password length and multi-character requirements
  • Enforce account disabling after an established number of login attempts
  • Use multi-factor authentication for accounts with sensitive data
  • Use the server or framework's session management controls only
  • Establish session inactivity timeout that is as short as possible
  • Session identifiers should only be located in the session cookie header
  • Do not disclose sensitive information in error responses
  • Error handlers should not display stack trace or debugging information
  • Users should not be able to download server-side source code
  • Avoid the use of known vulnerable functions
  • Do not pass user-supplied data to any dynamic execution function
  • Scan user uploaded files for viruses or malware
  • Explicitly initialize all your variables and other data stores, either during declaration or just before the first usage
